# Issue 本文テンプレート: feat（新機能追加）

`issue-create` が `type=feat`（ラベル `type:feature`）の Issue を起票する際に使う本文テンプレート。

## テンプレート本文

```markdown
## 概要

(何を、どこに対して、何をするか。対象モジュール / ディレクトリ / ファイルパスを明示。
 新機能名と提供価値を 1 文でまとめる)

## スコープ境界（feat 専用・必須）

本 Issue は **feat（新機能追加）の範囲内で閉じる** ものであり、以下の変更は **含まない**:

- bug 修正（別 Issue として起票する）
- refactor（測定可能な改善指標を持たない内部構造変更）
- docs のみの更新（コード変更を伴わないドキュメント改定）

`issue-review-ready` の観点 9（スコープ境界の明示）を満たすため、このセクションは **削除しないこと**。
やむを得ず混在させる場合は、理由を本セクションに明記する。

## 目的

### 現状の問題

(既存機能では何ができていないか。どのユースケースが満たされていないか。
 根拠となる 1 次情報: 既存 Issue、ユーザーからの要望ログ、関連ドキュメントのパス等)

### 到達したい状態

(このIssue完了後に何ができるようになるか。
 「[Role] として [Goal] のために [Action] したい」のユーザーストーリー形式を 1 本以上記載する
 — `issue-review-ready` 観点 8 を満たすため必須)

## 完了条件

- [ ] (客観的に判定可能な条件。例: 追加したコマンド / 関数が期待通りの出力を返す)
- [ ] (例: 対象コマンドが正しく動作する)
- [ ] 設計書作成（`draft/design/issue-<issue_id>-<slug>.md`）
- [ ] テスト作成（設計書のテスト戦略に従い S/M/L を網羅）
- [ ] 影響ドキュメントの更新（設計書「影響ドキュメント」セクション参照）
- [ ] `make check` 通過

## 参考

- 関連 Issue/PR: #xxx（あれば）
- 参考ドキュメント: docs/xxx
- (その他の 1 次情報 URL)
```

## 本文生成時のチェックポイント

`issue-review-ready` の **共通観点 1〜7 + feat 固有観点 8〜9** を満たすため、以下を対話で必ず収集する:

1. **対象領域の特定**（共通観点 2, 7）: 対象モジュール / ディレクトリ / ファイルパスの明示
2. **ユースケース / ユーザーストーリー**（feat 観点 8）: 「[Role] として [Goal] のために [Action] したい」形式を 1 本以上、または「誰が / どういう状況で / 何をしたくて / どう嬉しいか」の箇条書き。抽象的な「新機能を追加する」だけでは不可
3. **スコープ境界の宣言**（feat 観点 9）: bug 修正 / refactor / docs-only の混在を明示的に除外する記述。上記「スコープ境界」セクションを埋める
4. **完了条件の判定可能性**（共通観点 4）: 「〜が表示される」「〜がレスポンスに含まれる」等、CLI or 目視で判定可能な形
5. **1 次情報の明示**（共通観点 5）: 現状の問題の根拠となる既存 Issue / ログ / ドキュメントパス
6. **代替検討**: 既存機能での代替が難しい根拠（`issue-review-design` で深く問われる）

## 埋め方の補強ヒント

- 「便利にする」「改善する」等の主観表現は具体的なユースケースに置き換える
- 技術的な実装方針は Issue 本文には書かない（設計書で扱う）
- 「なぜ今やるか」のタイミング根拠があれば「目的」に追加（締め切り、依存 Issue 等）
