# Issue 本文テンプレート: bug（バグ修正）

`issue-create` が `type=fix`（ラベル `type:bug`）の Issue を起票する際に使う本文テンプレート。

## テンプレート本文

```markdown
## 概要

(どのモジュール / CLI コマンド / 機能で、何が期待と異なるかを 1 文で)

## 目的

### Observed Behavior（OB）

(観測された壊れた挙動。
 根拠: ログ / 再現したコマンド出力などを引用)

\`\`\`
(エラーログ、スタックトレース、出力結果の貼り付け)
\`\`\`

### Expected Behavior（EB）

(あるべき挙動。
 根拠: 仕様書パス / 既存テスト名 / ユーザー要求 Issue / ドキュメントパス)

### 再現手順（Steps to Reproduce）

1. 前提データ / 環境（環境変数、入力データ）
2. 実行コマンド / 操作手順
3. 観測される出力

## 完了条件

- [ ] 設計書で根本原因が特定されている
- [ ] 再現テストが 1 本以上追加され、修正前は FAIL、修正後は PASS することを確認（**実ログ代替**: 本文またはリンク先に OB を直接示す実世界障害ログが存在する場合は、実装前 FAIL ログの取得を省略し、修正後に回帰テストが PASS することの確認をもって可とする）
- [ ] 同根の他の壊れ箇所の調査結果が設計書に記載されている
- [ ] 影響モジュール全体のテストが green（デグレ無し）
- [ ] `make check` 通過
- [ ] (必要なら) 関連ドキュメントの更新

## 影響範囲（初期評価）

- 影響するモジュール / コマンド: (例: `src/starter_app/` 配下の CLI 実行パス全体)
- 深刻度: (例: データ欠損あり / 操作不能 / 表示の乱れ / 軽微)
- 回避策の有無: (例: なし / ワークフロー再実行で回復)

## 参考

- 関連 Issue/PR: #xxx（類似 bug、同一モジュールの過去修正）
- 関連するログ出力
- (その他の 1 次情報)
```

## 本文生成時のチェックポイント

bug Issue で review-ready を通すには以下が必須:

1. **OB の具体性**: エラーログ、コマンド出力のいずれかが貼られている
2. **EB の根拠**: 「こうあるべき」の根拠が 1 次情報（仕様書 / 既存テスト / Issue）で裏付けられている
3. **再現手順の完備**: 前提条件 + 操作 + 観測結果が番号付きで書かれている
4. **再現できない bug は Issue 化してはいけない**: 再現不能なら調査 Issue に格下げ。ただし本文またはリンク先に OB を直接示す実世界障害ログ（失敗コマンド・エラー文言・exit code・API 応答・関連 Issue/PR の実行ログ等）が存在し、**かつその OB に対応する EB を検証する恒久回帰テストが構築可能**な場合は admissible で、その実ログを実装前 Red 証跡の代替として扱える（合成再現の実装前 FAIL ログは省略可。修正後の回帰テスト Green は必須）。実ログがあっても回帰テストで OB を検証できない genuinely 再現不能な bug は調査 Issue に格下げする（実ログ代替は実装前 Red ログのみの例外であり、再現可能性そのものは免除しない）

## 埋め方の補強ヒント

- 「時々発生する」「環境によって違う」等の曖昧表現は調査 Issue として分離
- OB と EB を混ぜて書かない。別セクションで明確に分離
- 深刻度は運用インパクトで書く（技術的な難易度ではない）
