# Issue 本文テンプレート: docs（ドキュメント変更）

`issue-create` が `type=docs`（ラベル `type:docs`）の Issue を起票する際に使う本文テンプレート。

**本 Issue は docs-only workflow で進む**（`/i-doc-update` → `/i-doc-review` → `/i-doc-final-check` → `/i-pr`）。
コード変更を伴う場合はここでは扱わず、`feat` / `fix` / `refactor` のいずれかで起票すること。

## テンプレート本文

```markdown
## 概要

(対象ドキュメントのパスまたは領域を明示 + 何を変更するかを 1 文)

対象ドキュメント:
- `docs/xxx/yyy.md`
- (領域全体の場合) `docs/dev/` 配下の workflow 関連全ファイル

## 目的

### 現状の問題

(何が不足・誤り・古くなっているか。
 根拠: 実装コードとの乖離箇所、運用で混乱した事例、レビュー指摘履歴など)

### 到達したい状態

(修正後に何が明確になるか、誰がどう助かるか)

## 変更方針

- 事実整合: 実装・CLI・設定ファイルとドキュメントが一致する
- 実装整合: サンプルコマンド / コードが実際に動作する
- 運用整合: 運用手順が最新のスキル・スクリプト構成に準拠

## 完了条件

- [ ] 対象ドキュメントが現行の実装 / CLI / 運用方針と整合している
- [ ] リンクチェックで broken link がない
- [ ] docs-only workflow 内のレビュー（`/i-doc-review`）で PASS を得ている
- [ ] 関連する他のドキュメントにも波及更新の必要がないか確認済み

### ワークフロー完了後の確認項目

- なし

## スコープ外

- コード変更（コード変更を伴うなら別 Issue を feat / fix / refactor で起票）
- テスト追加（docs-only はテスト不要）

## 参考

- 関連 Issue/PR: #xxx
- 参照先ドキュメント: docs/xxx
- 実装箇所（docs と照合する対象）
```

## 本文生成時のチェックポイント

1. **対象ドキュメントのパスが明示されている**: 領域だけではなく具体パスがあると `i-doc-update` で迷わない
2. **コード変更が混ざらない宣言**: 「スコープ外」に明記する
3. **整合性の観点**: 事実・実装・運用の 3 整合性に言及
4. **事後確認の分離**（共通観点 14）: merge・実機適用・外部応答後など、workflow を RETRY しても環境非依存で同じ結果を得られない確認だけを末尾サブセクションへ移す。該当時は `- なし` を具体的な `- [ ]` 項目へ置換する

## 埋め方の補強ヒント

- 複数ドキュメントにまたがる横断的更新（スキル定義一括変更等）は、親 Issue + 子 Issue の構造を検討
- docs/adr/ を運用している場合、ADR の新規作成も `type=docs` で起票して良い
- コード変更と docs 更新が不可分な場合は `type=feat` / `type=fix` 側に寄せて docs 更新を完了条件に含める
