# Issue 本文テンプレート: refactor（リファクタリング）

`issue-create` が `type=refactor`（ラベル `type:refactor`）の Issue を起票する際に使う本文テンプレート。

## テンプレート本文

```markdown
## 概要

(対象モジュール / ファイル群と、リファクタの目的を 1 文で。
 例: 「ステップ実行モジュールを単一クラスから責務分離した 3 クラス構成に再編」)

## 目的

### 現状の問題（測定可能な形で）

(可読性・保守性・テスト性の問題を、客観的な観測値に落として記述。
 主観表現「コードが汚い」「改善したい」は禁止)

観測値の例:
- 循環依存: `import-linter` で N 件検出（該当出力を引用）
- 重複コード: grep `<pattern>` で同一ロジックが N 箇所
- テスト不可能性: mock 注入点が取れず単体テストが書けない具体例
- 実行時間: ベンチマーク結果（`pytest --benchmark` 等）
- 対象モジュールのカバレッジ: X%

### 改善指標（測定可能な目標）

現状の問題に 1 対 1 対応する定量 or 定性の目標を設定:
- (例) 循環依存 0 件
- (例) 重複箇所を 1 箇所に統合、N 行削減
- (例) 対象関数の Small テストが書ける（現状不可）
- (例) 実行時間 X% 削減

## 完了条件

- [ ] ベースライン計測結果が設計書に記録されている
- [ ] 改修後の再計測で改善指標を達成している
- [ ] 既存テストがすべて PASS（振る舞い非変更の証明）
- [ ] 必要に応じて safety net テストが追加されている
- [ ] 公開 IF に変更がない（あるなら設計書で承認済み）
- [ ] `make check` 通過

### ワークフロー完了後の確認項目

- なし

## 対象スコープの明示

- 対象ファイル / ディレクトリ: (例: `src/starter_app/example.py` および関連モジュール)
- 公開 IF の扱い: 不変 / 変更あり（変更ありなら影響範囲）
- 混在禁止: feat / bug 修正を混ぜない（混ざる場合は別 Issue）
- docs のみの更新との混在禁止: docs 変更を伴う場合も明記する

## 参考

- 関連 Issue/PR: #xxx
- 参考リファクタ手法 / 設計パターン
- 対象コードの該当箇所（ファイルパス:行）
```

## 本文生成時のチェックポイント

1. **現状の問題が観測可能**: 「汚い」「わかりにくい」等の主観ではなく、ツール出力 / 計測値に落ちている
2. **改善指標が定量 or 定性で固定**: 完了判定に使える形
3. **対象スコープの明示**: 対象モジュール / ディレクトリ / ファイルパスと「混在禁止」の宣言
4. **振る舞い非変更の絶対性**: 「ついでに」の一言も許容しない
5. **事後確認の分離**（共通観点 14）: merge・実機適用・外部応答後など、workflow を RETRY しても環境非依存で同じ結果を得られない確認だけを末尾サブセクションへ移す。該当時は `- なし` を具体的な `- [ ]` 項目へ置換する

## 埋め方の補強ヒント

- パフォーマンス refactor の場合、ベンチ結果（before 値）を Issue 作成時点で取れるなら本文に貼る
- 公開 IF 変更が不可避な場合は feat / fix に分類した方が適切ではないか再検討
- 大きな refactor は段階 Issue に分割（例: 抽出 → 置換 → 削除 を別 PR）
