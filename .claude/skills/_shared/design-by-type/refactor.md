# Design by Type: refactor（リファクタリング）

`type:refactor` ラベルが付いた Issue の設計書を書く際の構成指針。

## refactor 設計の性格

- **外部から観測可能な振る舞いを変えない**ことが絶対要件。
- 「何を良くするか」を**測定可能な指標**で固定しないと、作業が美学の押し付けになる。
- テストは既存のものを流用する前提。新規テストは原則「振る舞い非変更」を保証する bridging test のみ。

## 必須セクション

### 1. 概要（1〜2 文）

- 対象モジュール + リファクタの目的（何を良くするか）を 1 文。
- 例: 「`starter_app.core` を単一モジュールから責務分離した 3 モジュール構成に再編。循環依存を解消する」

### 2. 背景・目的（改善指標必須）

- **現状の問題**: 何が辛いか。「可読性が低い」等の主観は不可。以下のような**観測可能な形**に落とす:
  - 循環依存の存在（静的解析ツール出力）
  - 同一ロジックの重複箇所数（grep 結果）
  - テスト不可能性（mock 注入点が取れない具体例）
  - 実行時間 / メモリ消費（ベンチマーク結果）
- **改善指標**: 上記に対応する定量 or 定性のゴール:
  - 循環依存 0 件（検出ツールのレポート）
  - 重複箇所統合で N 行削減
  - 対象関数の単体テストが書ける（現状不可）
  - 実行時間 X% 削減

### 3. ベースライン計測

改修前の状態を記録する。実装フェーズの冒頭で再計測し、改修後と比較するための基準:

- コード行数 / 循環的複雑度
- テストカバレッジ（変更対象モジュール）
- ベンチマーク結果（パフォーマンス refactor の場合）
- 静的解析ツール（ruff / mypy）の警告数

計測コマンドは設計書に明記し、誰が実行しても同じ値が出る形にする。

### 4. インターフェース

- **公開 IF は原則不変**を宣言
- 不可避に変更が出る場合は、変更箇所と移行パスを列挙（deprecation 期間など）
- 内部 IF（モジュール間の呼び出し）の新構造を簡潔に提示

### 5. 変更スコープ

- 影響するファイル群を明示
- scope が広がりそうなら段階分割を検討（「最初の PR は抽出のみ、第 2 PR で置換」等）
- 本プロジェクトは Python 単一スタック。Scope 分岐は扱わない

### 6. 方針

- Before / After の構造を図 or 擬似コードで併記
- 移行ステップ（「A → B の順で置換」等の順序）

### 7. テスト戦略（振る舞い非変更の保証）

**refactor 固有ルール**: 振る舞いが変わっていないことを既存テストで担保できるか事前に評価する。
詳細は [docs/dev/testing-convention.md](../../../../docs/dev/testing-convention.md) を参照。

- **既存テストのカバレッジ評価**:
  - 変更対象モジュールのテストカバレッジを計測
  - カバレッジが低い箇所は **先にテストを足してから refactor に入る**（safety net）
- **bridging test**:
  - リファクタ前後で同じ入力 → 同じ出力を保証するテストを 1 本以上
  - 既存テストで十分な場合は新規不要（エビデンスを書く）
- **性能 refactor の場合**: ベースライン vs 改修後のベンチを Large テストに含める

### 8. 影響ドキュメント

- `docs/adr/`: 内部構造の意思決定がある場合は ADR を新設（`docs/adr/` を運用している場合）
- `docs/dev/`: 開発ワークフローやテスト規約に影響する場合
- `docs/reference/python-standards.md`: コーディング規約に影響する場合
- 外部向けドキュメント: IF 不変なら更新不要、変更があれば必須

### 9. 参照情報

- リファクタ手法の根拠（Fowler の Refactoring 書、内部 RFC 等があれば）
- 既存コードの該当箇所（ファイルパス:行）

## よくある指摘

- 「可読性向上」等の主観目標だけで測定指標がない → CR
- ベースライン計測がなく、改善の実証ができない → CR
- 振る舞い変更が混入（ついでに仕様変更をしている） → CR + Scope 分離要求
- 既存テストのカバレッジ不足を放置して refactor 着手 → safety net 構築を先行
