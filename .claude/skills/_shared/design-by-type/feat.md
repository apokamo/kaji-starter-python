# Design by Type: feat（新機能追加）

`type:feature` ラベルが付いた Issue の設計書を書く際の構成指針。
本ファイルは `issue-design` スキルから dispatch される。

## feat 設計の性格

- **新しい価値**を足すのが主目的。既存動作に対する制約より、**ユースケースと IF 設計**が中心。
- "観測された挙動が間違っている" という前提がない → bug 用の OB/EB セクションは不要。
- 代わりに「誰が」「どの操作で」「何を得るか」をユースケースで固定する。

## 必須セクション

`issue-design` の設計書テンプレートを下地に、feat では以下を**具体化**する。

### 1. 概要（1〜2 文）

- 新機能名 + 提供価値を 1 文。
- 例: 「CLI の `--json` 出力オプション。結果を機械可読形式で取得できるようにする」

### 2. 背景・目的

- **ユーザーストーリー or ユースケース**を 1 本以上:
  「[Role] として、[Goal] のために、[Action] したい」
- 代替案と不採用理由（任意だが推奨）: 既存機能での代替が難しい根拠を書くと review-design が通りやすい。

### 3. インターフェース

feat は IF が契約そのもの。以下を必ず埋める。

- **入力**: 引数・パラメータ・設定（型・単位・必須/任意）
- **出力**: 戻り値・副作用（ファイル書き込み / CLI 出力 / ログ / 外部 API リクエスト）・生成物
- **使用例**: 呼び出し側のコードを 3〜10 行の擬似コードで提示
- **エラー**: 想定される失敗（バリデーションエラー、外部依存失敗）と、その際の戻り値・例外

### 4. 制約・前提条件

- 既存モジュールへの依存（import 先、ファイル I/O、外部 API、サブプロセス）
- パフォーマンス制約（応答時間、同時実行数、レート制限）
- 既存機能との互換性保持ルール

### 5. 変更スコープ

- 影響するモジュール・ファイル群を明示
- 本プロジェクトは Python 単一スタック。backend / frontend / fullstack の Scope 分岐は扱わない

### 6. 方針（Minimal How）

- 主要なデータフローを文章 or シーケンスで記述
- 新規に作るモジュール・クラス・関数の**名前と責務**のみ（実装詳細は書かない）

### 7. テスト戦略

feat は「契約（IF）と価値（ユースケース）の双方」を検証する。
[docs/dev/testing-convention.md](../../../../docs/dev/testing-convention.md) のサイズ定義に従い、以下を検討する:

- **Small**: バリデーション、マッピング、純粋関数
- **Medium**: ファイル I/O / サブプロセス / 内部サービス結合
- **Large**: 実 API 疎通や E2E フロー（CLI を実際に起動するケース等）

> **省略判断**: [docs/dev/testing-convention.md](../../../../docs/dev/testing-convention.md) の「省略してよい理由 / 省略してはいけない理由」にある 4 条件をすべて満たす場合のみスキップ可。実行時コード変更を伴う feat で「Small で十分」と決め打ちしない。

### 8. 影響ドキュメント

feat は新しい利用者向け文書が発生しやすい。以下を必ず評価:

- `README.md` / `docs/README.md` — 利用者向け説明・索引の追加
- `docs/reference/python-standards.md` — コーディング規約に影響する場合
- `docs/dev/` — 開発ワークフロー・テスト規約への影響
- `docs/adr/` — 技術選定（新ライブラリ採用等）があれば必須（`docs/adr/` を運用している場合）
- `AGENTS.md` / `CLAUDE.md` — プロジェクト規約や必読ドキュメントの更新

### 9. 参照情報（Primary Sources）

- 新規採用ライブラリ / API / プロトコルの公式ドキュメント URL
- 設計判断の根拠となる引用を 1〜2 行で添える

## よくある指摘（review-design で CR になりやすい）

- IF の型・単位が曖昧（「数値」としか書いていない）
- 使用例がない / 実装詳細を書きすぎて What を超えている
- エラーケースの挙動が未定義
- Medium / Large テストを「Small で十分」と言って省略（`testing-convention` の 4 条件を満たさないのに省略している）

## docs type の扱い

type=docs の Issue は本スキルの対象外。`/i-doc-update` にルーティングすること。
