# Critical Decision Checklist（正本）

workflow 系 skill が共有する、人間の重要判断と AI の仮定を分離するための正本。
`issue-review-ready` / `issue-design` / `issue-review-design` は本書を参照し、
判定基準を個別の skill に複製しない。

workflow 起動前に要件を聞き取る `grill-me` も、本書の判断軸と provenance 形式を
再利用する。本書を質問文の固定リストとして複製せず、Issue と調査結果から未決 branch を
組み立てる。

## 原則

workflow 開始後は、原則として人間の同期介入なしに後続 step が進む。したがって、
人間が決めた重要判断を AI が上書き、弱化、または「参考資料」へ格下げしてはならない。
一次情報同士が矛盾する場合も、AI が都合のよい方を選ばない。

`issue-design` は、要件や重要方針を新たに決める場ではなく、**決定済み方針を実装可能な
粒度へ詳細化する場**である。決定範囲内の実装方針、テスト観点、既存パターンの適用は
AI が具体化してよい。

## 重要判断の 3 分類

許可事項の whitelist ではなく、**判断を間違えたときに後段で安く直せるか（可逆性）**
で分類する。

| 分類 | 状態 | 扱い | 記録 |
|------|------|------|------|
| 1. 決定済み方針の詳細化 | 人間が方針と決定範囲を示している | 決定範囲内で進む | 人間決定の出典と、設計で行った詳細化を記録 |
| 2. two-way door の未決 | 未決だが、誤っても review / verify / PR review で安く直せる | 仮定と根拠を明記して進む | AI の仮定であることと後段の検査先を記録 |
| 3. one-way door の未決 | 不可逆、または後段での修正コストが高い | AI が補完せず `ABORT` | `suggestion` に人間が決める項目を具体的に列挙 |

分類 2 は AI が無記録で決定してよいという意味ではない。仮定を人間決定に見せかけず、
後段で再検査できる形にする。分類 2 と 3 の境界に迷う場合は分類 3 として停止する。

## source of truth の扱い

- Issue 本文、Issue コメント、既存契約、mock、Figma、仕様書などについて、人間が
  source of truth と明示した指定を保持する
- source of truth を変更・弱化・参考資料へ格下げするには、人間の明示的な確認を要する
- 複数の source of truth が矛盾し、優先順位が人間によって決まっていない場合は
  one-way door の未決として停止する
- AI が作成した要約や設計書は、それだけを理由に人間決定の出典として扱わない。
  人間が承認・指定した事実を辿れる場合のみ、その参照先を provenance に記録する
- 後の人間コメントが先の決定を明示的に変更した場合は、最新の決定と変更元の両方を
  辿れるようにする。暗黙の上書きはしない

## one-way door になりやすい判断軸

以下は代表例であり、完全な列挙ではない。項目名ではなく可逆性で最終判定する。

### ユーザー価値・UI/UX

- 誰のどの課題を解決するか、主要ユースケース、画面・操作構造
- mock / Figma / screenshot / prototype を source of truth とするか
- 既存 UI を維持する範囲と、再構築する範囲
- スコープ外の差分を今回取り込むか

### 公開 CLI / API とデータ契約

- 公開 CLI の引数、終了コード、標準出力・標準エラーの契約
- 公開 API の request / response、エラー、互換性の契約
- 永続化 schema、ファイル形式、イベント形式の変更
- migration、既存データ移行、rollback 方針

### 運用・互換性

- deploy、batch、scheduler、監視、権限、障害対応などの運用影響
- 既存利用者、外部連携、既存データに対する非互換変更
- quality gate や必要なテストの省略
- Issue の目的や完了条件を変えるスコープ変更

命名、内部関数の分割、局所的なログ文言など、後段で安く直せる詳細は通常
two-way door である。ただし公開契約や運用手順に露出する場合は同じ扱いにしない。

## provenance の記録形式

設計書には、重要判断ごとに最低限次の 4 列を持つ表を置く。

| 判断 | 方針 | 出典または仮定 | 設計で行った詳細化 |
|------|------|----------------|--------------------|
| 公開 CLI の互換性 | 既存引数を維持する | Issue 本文「変更方針」（人間決定） | 新オプションだけを追加する構成へ分解 |
| page size | 50 件 | AI の仮定。既存一覧の慣習が根拠。review-code で検査 | default 値として定義 |

記録時は次を満たす。

- **判断**: 何を決めたかを、後段で独立に検査できる単位で書く
- **方針**: 採用する内容を書く
- **出典または仮定**: 人間決定なら Issue 本文の節、コメント、既存契約など検証可能な
  参照を示す。AI の仮定なら「AI の仮定」、根拠、後段の検査先を明記する
- **設計で行った詳細化**: 人間決定そのものと、AI が決定範囲内で具体化した内容を分ける
- 判断対象がない場合は表を省略せず、「該当なし」と、その確認根拠を記録する

provenance 欄が空、出典を検証できない、または AI が one-way door を自己解釈で
埋めている場合は設計を進めない。

## step ごとの防御

| step | 責務 |
|------|------|
| `grill-me`（workflow 外・明示起動） | 有人区間で未決 branch を 1 問ずつ確認し、人間決定を Issue 本文 `## 決定事項`、根拠を provenance コメントへ固定する。readiness 判定は行わない |
| `issue-review-ready` | Issue 本文と人間コメントから one-way door の未決・source of truth の矛盾を着手前に検出する |
| `issue-design` | 決定済み方針を詳細化し、判断 provenance と AI の仮定を設計書へ記録する。設計中に one-way door が顕在化したら停止する |
| `issue-review-design` | provenance の出典を独立に確認し、人間決定の上書き、仮定の偽装、未決の one-way door がないか検査する |

早い step で止めるほど手戻りが小さいが、前段を通過したことを理由に後段の検査を
省略しない。

## verdict の使い分け

- **記述・反映不備**: 人間の決定は存在し、参照先や provenance の書き方を補う、
  または明示済みの方針へ設計を戻せば直せる場合は `RETRY`
- **one-way door の真の未決**: 人間の決定自体が存在しない、source of truth が矛盾する、
  または AI が人間の判断を代行しなければ進めない場合は `ABORT`
- `ABORT` の `suggestion` には「誰かに確認する」のような抽象表現ではなく、決めるべき
  項目、競合する選択肢・情報源、再開条件を列挙する

人間決定の上書き・弱化・格下げは必ず指摘するが、元の決定が明確で機械的に復元できる
場合は `RETRY` でよい。`RETRY` を使って AI の fix step に one-way door を埋めさせては
ならない。既存の
`PASS` / `RETRY` / `ABORT` で表現できるため、新しい verdict や workflow 遷移は
追加しない。

## 非目標

- mock や Figma を常に source of truth とすること
- Issue に書かれた事項しか AI が詳細化できない whitelist 制約
- one-way door の全項目を固定的に列挙すること
- 既存 Issue や設計書を一括移行すること
