# Testing Convention

テストの分類・命名・追加判断の規約。

## テストサイズ定義

| サイズ | marker | 特徴 | 例 |
|--------|--------|------|-----|
| Small | `small` | 外部依存なし。純粋ロジック・単体関数 | 関数の入出力検証 |
| Medium | `medium` | ファイル I/O・mock した外部プロセス | 設定ファイル読み込み |
| Large | `large` | 実 CLI・E2E・外部サービス | CLI 実行の統合確認 |

```bash
make test          # 全マーカー
make test-small    # pytest -m small
make test-medium   # pytest -m medium
make test-large    # pytest -m large
```

## テストファイル命名規約

- `tests/test_<module>.py` を基本とする
- テスト関数は `test_<対象>_<条件>_<期待>` を目安に、日本語 docstring で意図を補う

## テスト戦略の原則

- 新規ロジックには原則 Small テストを付ける（TDD-first）
- 外部依存は Small では持ち込まない。必要なら Medium に分離する
- Large は「1 周通ることの確認」に限定し、細部の分岐は Small / Medium で覆う

### 変更タイプごとの期待値

| 変更タイプ | 期待されるテスト |
|-----------|----------------|
| 実行時コード変更 | 変更ロジックを覆う Small（必要に応じ Medium / Large） |
| docs-only | 恒久テスト不要。`make verify-docs` が gate |
| 設定変更 | 恒久テスト不要。`uv run kaji validate` 等の決定的検証で代替 |

## 恒久回帰テストと変更固有検証

- **恒久回帰テスト**: `tests/` に残り、以後の全変更で回る。将来の回帰を検出する価値がある場合に追加する
- **変更固有の一時検証**: その変更の正しさを一度だけ確認する検証。結果を Issue コメントに証跡として残し、`tests/` には残さない

### 恒久テストを追加しない場合の 4 条件

恒久テストを追加しない判断は、以下の 4 条件で説明できる場合のみ正当化される:

1. 独自ロジックの追加・変更をほぼ含まない
2. 想定される不具合パターンが既存 gate（`make check` / `uv run kaji validate` / `make verify-docs` 等）で捕捉済み
3. 新規テストを追加しても回帰検出情報がほとんど増えない
4. 上記理由をレビュー可能な形（設計書・Issue コメント）で説明している

### 不正当な省略理由（レビューで Changes Requested となる）

- 「時間がないため」「簡単な変更のため」だけの省略
- 失敗しうる分岐・エラーパスがあるのにテストがない
- 既存テストの失敗を放置したままの追加変更

## 設計書テスト戦略セクションの書き方

設計書には変更タイプを明示し、実行時コード変更なら Small / Medium / Large の
検証観点を、docs-only / 設定変更なら変更固有検証と「恒久テストを追加しない理由」
（上記 4 条件に沿って）を記載する。

## レビュー時のチェックリスト

- [ ] 変更タイプ（実行時コード変更 / docs-only / 設定変更）が明示されているか
- [ ] 実行時コード変更なら Small / Medium / Large の検証観点が定義されているか
- [ ] 恒久テストを追加しない場合、その理由が 4 条件に沿って説明されているか
